# i18n Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Vietnamese (default) and English locales to device-hub using `next-intl` with cookie-based locale resolution. Every existing user-facing string is translated. Users switch locale from the topbar user menu and from the login page.

**Architecture:** `next-intl` resolves the locale on the server from a `NEXT_LOCALE` cookie via `getRequestConfig`. The root layout wraps the app in `NextIntlClientProvider`. A client-side `setLocale()` helper writes the cookie and calls `router.refresh()`. Messages live in `client/src/messages/<locale>.json`, one file per locale, top-level namespaced. No client store — the cookie is the source of truth and `useLocale()` exposes it to React.

**Tech Stack:** Next.js 16 (App Router), `next-intl`, React 19, TypeScript, shadcn/ui dropdown for the topbar switcher.

**Spec:** `docs/superpowers/specs/2026-06-06-i18n-design.md`

**Working directory:** all commands assume `cd client/` unless stated otherwise.

**Verification:** the project has no test harness yet. Each task's verification step is `pnpm tsc --noEmit && pnpm lint` plus manual UI smoke-test when relevant. Commit at the end of each task.

```ts
export const SUPPORTED_LOCALES = ["vi", "en"] as const;
export type Locale = (typeof SUPPORTED_LOCALES)[number];

export const DEFAULT_LOCALE: Locale = "vi";

export const LOCALE_COOKIE = "NEXT_LOCALE";

  vi: "Tiếng Việt",
  en: "English",
};

export function isLocale(value: unknown): value is Locale {
  return typeof value === "string" && (SUPPORTED_LOCALES as readonly string[]).includes(value);
}
```

```ts
// client/src/i18n/request.ts
import { cookies } from "next/headers";
import { getRequestConfig } from "next-intl/server";
import { DEFAULT_LOCALE, LOCALE_COOKIE, isLocale } from "@/lib/i18n/locales";

export default getRequestConfig(async () => {
  const cookieStore = await cookies();
  const candidate = cookieStore.get(LOCALE_COOKIE)?.value;
  const locale = isLocale(candidate) ? candidate : DEFAULT_LOCALE;
  const messages = (await import(`@/messages/${locale}.json`)).default;
  return { locale, messages };
});
```

## Task 5: Create the setLocale client helper

**Files:**
- Create: `client/src/lib/i18n/set-locale.ts`

- [ ] **Step 1: Write the helper**

```ts
// client/src/lib/i18n/set-locale.ts
"use client";

import { LOCALE_COOKIE, isLocale, type Locale } from "@/lib/i18n/locales";

// Writes the locale cookie and reloads route data so server components
// re-render with the new locale's messages.
export function setLocale(next: Locale, refresh: () => void): void {
  if (!isLocale(next)) return;
  // 1 year, site-wide, Lax — locale isn't sensitive enough for Strict.
  document.cookie = `${LOCALE_COOKIE}=${next}; path=/; max-age=${60 * 60 * 24 * 365}; SameSite=Lax`;
  refresh();
}
```

- [ ] **Step 1: Write the component**

A dropdown in the topbar (button shows a globe icon and the active locale label). Uses the existing shadcn `DropdownMenu`.

```tsx
// client/src/components/app/language-switcher.tsx
"use client";

import { Check, Globe } from "lucide-react";
import { useLocale } from "next-intl";
import { useRouter } from "next/navigation";
import { useTransition } from "react";

import { Button } from "@/components/ui/button";
import {
  DropdownMenu,
  DropdownMenuContent,
  DropdownMenuItem,
  DropdownMenuTrigger,
} from "@/components/ui/dropdown-menu";
import { LOCALE_LABELS, SUPPORTED_LOCALES, isLocale, type Locale } from "@/lib/i18n/locales";
import { setLocale } from "@/lib/i18n/set-locale";
import { cn } from "@/lib/utils";

export function LanguageSwitcher() {
  const active = useLocale();
  const router = useRouter();
  const [, startTransition] = useTransition();

  function choose(next: Locale) {
    setLocale(next, () => startTransition(() => router.refresh()));
  }

  return (
    <DropdownMenu>
      <DropdownMenuTrigger asChild>
        <Button variant="ghost" size="sm" className="gap-2">
          <Globe className="size-4" aria-hidden />
          <span className="text-sm">
            {isLocale(active) ? LOCALE_LABELS[active] : active}
          </span>
        </Button>
      </DropdownMenuTrigger>
      <DropdownMenuContent align="end" className="min-w-[160px]">
        {SUPPORTED_LOCALES.map((loc) => (
          <DropdownMenuItem
            key={loc}
            onSelect={() => choose(loc)}
            className="flex items-center justify-between"
          >
            <span>{LOCALE_LABELS[loc]}</span>
            <Check className={cn("size-4", loc === active ? "opacity-100" : "opacity-0")} aria-hidden />
          </DropdownMenuItem>
        ))}
      </DropdownMenuContent>
    </DropdownMenu>
  );
}
```

- [ ] **Step 3: Rewrite the login page**

```tsx
    });

        {loading ? t("redirecting") : t("continueWithGoogle")}
      </Button>
```

## Notes for the implementer

- **Vietnamese quality**: drafts in this plan come from Claude. A native speaker should review later. If you (the engineer) are Vietnamese and disagree with a phrasing, change the JSON — the keys are stable.
- **Key drift between locales**: when you add a key to one locale file, add it to the other in the same commit. next-intl will hard-fail at runtime if a key is missing in the active locale.
- **Server vs client**: `getTranslations` for server components and server actions; `useTranslations` for client components. They share the same JSON.
- **Domain values vs labels**: enum *keys* like `"in-use"` stay in the DB and in code. Only the *displayed labels* are translated through the message files.
- **What's intentionally untranslated**: form validation messages produced by Zod (`zod` defaults), Supabase auth error strings, the metadata `title` / `description` in the root layout, brand wordmark glyph. All of these can be migrated later if you want.
