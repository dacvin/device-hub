# Internationalization (i18n) — Design

**Date:** 2026-06-06
**Status:** Approved, ready for implementation plan
**Author:** Vinh Huynh + Claude

## Goal

Add multi-language support to device-hub with Vietnamese (`vi`) as the default locale and English (`en`) as the alternate. Every existing user-facing string is translated. Users switch locale via a language switcher that appears both in the post-login user menu and on the login page.

## Non-goals

- URL-based locales (no `/vi/...` or `/en/...` prefixes)
- Zustand or any client store for locale state (cookie + `useLocale()` is sufficient)
- Translating database content (device names, department names, etc.) — stays as stored
- Translating Supabase email templates or server-side error messages owned by Supabase
- Custom date/number/currency formatting beyond what next-intl provides by default
- Adding a test harness (project has none yet); verification is manual

## Decisions

| Decision | Choice | Reason |
| --- | --- | --- |
| Library | `next-intl` | De-facto standard for Next.js App Router; works in server + client components; supports cookie-based locale resolution without URL routing |
| Locale source | Cookie `NEXT_LOCALE` | Readable from both server (`next/headers`) and client (`document.cookie`); no URL clutter; no hydration flash |
| Default locale | `vi` | Project-level requirement |
| Supported locales | `vi`, `en` | |
| State management | None beyond next-intl + cookie | The cookie is the single source of truth; `useLocale()` exposes it to any client component. Zustand was considered and dropped as redundant. |
| Message format | One JSON file per locale, top-level namespaced | Easy to scan, one file per locale to keep in sync, translator-friendly |
| Switcher placement | Avatar dropdown in topbar **and** login page | Login switcher lets a user pick a language before signing in |

## Architecture

### Locale resolution flow

1. Browser sends request with `NEXT_LOCALE` cookie (or none).
2. `src/i18n/request.ts` runs via `getRequestConfig`. It reads the cookie via `cookies()` from `next/headers`, validates it against `SUPPORTED_LOCALES`, and falls back to `vi` if missing or invalid.
3. It returns `{ locale, messages }` where `messages` is the dynamically imported JSON for that locale.
4. `<NextIntlClientProvider locale={locale} messages={messages}>` wraps the app in `app/layout.tsx`.
5. Server components call `await getTranslations(ns)`; client components call `useTranslations(ns)`.
6. `<html lang={locale}>` is set in the root layout so the browser/assistive tech knows the language.

### Switching locale

1. User clicks a language item in either the user dropdown or the login-page switcher.
2. A client helper `setLocale(next)` runs:
   - Validates `next` against `SUPPORTED_LOCALES`.
   - Writes `document.cookie = "NEXT_LOCALE=<next>; path=/; max-age=31536000; SameSite=Lax"`.
   - Calls `router.refresh()` so the server re-resolves messages from the new cookie value.
3. The whole app re-renders in the new locale. The cookie persists across reloads (1-year max-age).

## File layout

```
client/
├── next.config.ts                          # wrap with createNextIntlPlugin
└── src/
    ├── i18n/
    │   └── request.ts                      # getRequestConfig — reads cookie, returns { locale, messages }
    ├── messages/
    │   ├── vi.json
    │   └── en.json
    ├── lib/
    │   └── i18n/
    │       ├── locales.ts                  # SUPPORTED_LOCALES, DEFAULT_LOCALE, isLocale(), LOCALE_LABELS
    │       └── set-locale.ts               # client helper: validate, write cookie, router.refresh()
    ├── app/
    │   ├── layout.tsx                      # NextIntlClientProvider + <html lang>
    │   └── login/
    │       └── _components/
    │           └── login-language-switcher.tsx  # small switcher on login page
    └── components/
        └── app/
            └── language-switcher.tsx       # used inside the user menu in topbar
```

## Message file structure

`messages/vi.json` and `messages/en.json` share an identical key shape. New strings must be added to **both** files in the same commit.

```json
{
  "common":        { "save", "cancel", "delete", "edit", "search", "loading", "back", ... },
  "login":         { "title", "subtitle", "continueWithGoogle", "termsPrefix", "termsLink" },
  "nav":           { "devices", "departments", "groups", "manufacturers" },
  "topbar":        { "userMenu.signOut", "userMenu.account", ... },
  "devices":       { "list.title", "form.fields.*", "details.*", ... },
  "departments":   { ... },
  "groups":        { ... },
  "manufacturers": { ... },
  "settings":      { "language", "languageLabel.vi", "languageLabel.en" }
}
```

Namespaces map to features under `app/(app)/<feature>/`. Shared UI strings live in `common`.

## Language switcher UX

### In the topbar user menu (post-login)
Inside the avatar dropdown, a section showing:
- Tiếng Việt
- English

Active locale shows a check mark. Click → `setLocale(...)`.

### On the login page (pre-login)
A small inline switcher in a corner of the left pane (the same pane that holds the sign-in form). Two-state visual (e.g., text buttons "Tiếng Việt · English" with the active one highlighted). Click → `setLocale(...)`. The page re-renders in the new locale.

## Migration of existing strings

Every user-facing string in the following files is replaced with `t("key")` (server `getTranslations`) or `useTranslations` (client):

- `app/login/page.tsx`
- `app/login/_components/login-form.tsx`
- `app/(app)/layout.tsx`
- `components/app/topbar.tsx`

The implementation plan enumerates each string and its key.

## Validation / testing

Manual only:
1. Clear cookies, load app → renders in Vietnamese (default).
2. Sign in → switch to English in user menu → entire app re-renders in English.
3. Reload → still English (cookie persisted).
4. Sign out → back at login page in English; switch to Vietnamese in login-page switcher → login UI in Vietnamese.
5. Inspect cookie in DevTools to confirm `NEXT_LOCALE` is set with `max-age` ~1 year.
6. View source on first SSR render to confirm `<html lang>` matches the cookie.

## Risks and open questions

- **Translation quality**: Vietnamese strings will be drafted by Claude. A native-speaker review pass is out of scope but recommended before this reaches end users.
- **String drift**: New code may forget to add keys to both locale files. Mitigate via a follow-up lint rule or type generation; out of scope here.
- **Login page first paint**: The login page is the entry point. If a user has never visited, default `vi` is used. This is intended behavior.
